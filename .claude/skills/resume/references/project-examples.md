## 范例 1:企业级智能客服 Agent

**项目名称**:企业知识库智能客服系统
**项目周期**:2025.03 - 2025.11 | **角色**:Agent 后端负责人(3 人团队)

**项目背景**:
公司客服团队每日处理 8000+ 工单,60% 为重复性问题,人工成本高且响应慢(平均 12 分钟)。
需要构建基于公司知识库的 7×24 智能客服 Agent,目标自动解决 70% 工单。

**技术方案**:
- 基于 **LangGraph** 设计状态机式 Agent,包含意图识别、检索、生成、转人工 4 个节点
- RAG 系统使用 **bge-large-zh + Qdrant** 做向量检索,**bge-reranker-v2** 二次排序,Top-5 召回准确率从 68% 提升到 91%
- 引入 **HyDE(假设性文档嵌入)** 处理短查询,长尾问题召回率提升 23%
- 设计 **三级置信度评估**:LLM-as-Judge + 规则校验 + 用户反馈,低置信度自动转人工,误转人工率控制在 5% 以内
- 通过 **Prompt Caching**(Claude Sonnet 4.6)将系统提示词缓存,推理成本降低 64%

**个人贡献**:
- 主导 Agent 整体架构设计与 LangGraph 状态机编排
- 设计并实现混合检索(BM25 + 向量 + 知识图谱),解决专有名词召回差的问题
- 搭建 **LangFuse** 可观测性平台,追踪 token 消耗、延迟、用户满意度

**项目成果**:
- 上线 3 个月覆盖 12 个业务线,日均处理 6500 工单,自动解决率 73%
- 客服团队规模从 25 人缩减到 9 人,年度人力成本节省 ¥320W
- 平均响应时间从 12 分钟降到 8 秒,NPS 提升 18 分
- 系统稳定运行,P99 延迟 < 3s,可用性 99.95%

## 范例 2:多 Agent 协作的代码评审系统

**项目名称**:基于多 Agent 的代码安全评审平台
**项目周期**:2025.08 - 2026.02 | **角色**:核心开发(5 人团队)

**项目背景**:
公司每周 200+ PR,人工 Code Review 覆盖率不足 40%,安全漏洞滞后发现造成多次线上事故。
需构建自动化 Review Agent,在 PR 创建后自动评审并给出可执行的修复建议。

**技术方案**:
- 基于 **Anthropic Agent SDK + MCP** 构建,4 个专家 Agent 并行评审:
  - SecurityAgent(基于 OWASP Top 10 知识库 + CodeQL MCP)
  - PerformanceAgent(基于历史性能数据 + 火焰图分析)
  - StyleAgent(基于项目 ESLint/Ruff 规则)
  - LogicAgent(基于业务领域知识库)
- 由 **OrchestratorAgent** 汇总各 Agent 结论,通过 Constitutional AI 方法过滤误报
- 使用 **GitHub MCP Server** 直接发布 Inline Review 评论
- 评估体系:基于 **Promptfoo** 搭建 500+ 真实 PR 的 golden set,持续监控召回率与精确率

**个人贡献**:
- 设计多 Agent 通信协议与冲突解决机制
- 实现自研 MCP Server(对接公司内部知识库与 CI 系统),开源版本 GitHub 380+ stars
- 主导评估体系建设,误报率从首版 32% 降到 7%

**项目成果**:
- 覆盖公司 80% 仓库,平均每个 PR 产出 4.2 条有效建议
- 上线后线上 P0 安全事故归零(此前每季度 2-3 起)
- 平均 Code Review 周期从 2.3 天缩短到 4 小时
- 团队相关论文中稿 EMNLP 2025 Industry Track

## 范例 3:Coding Agent / Browser Agent(2026 热门)

**项目名称**:面向数据分析师的浏览器自动化 Agent
**项目周期**:2025.12 - 2026.05 | **角色**:技术 Owner(2 人小组)

**项目背景**:
数据分析师 30% 工时耗在跨多个 SaaS 平台(GA、Mixpanel、Tableau)取数与拼数据,机械重复。
需要一个能听懂自然语言、自动操作浏览器完成跨平台取数的 Agent。

**技术方案**:
- 基于 **Claude Computer Use + Playwright MCP** 构建浏览器操作 Agent
- 设计 **任务分解 + 反思** 循环:先用 Claude Opus 4.7 做任务规划,再由 Sonnet 4.6 逐步执行,失败时触发反思重试
- **视觉理解 + DOM 双通道**:截图理解页面布局,DOM 提取精确元素,降低误点击率
- 引入 **Episodic Memory**(基于 Mem0),Agent 学习过的操作流程持久化,二次执行从 8 分钟降到 90 秒
- 安全机制:所有写操作(点击"删除"、"导出")必须人工确认

**个人贡献**:
- 端到端架构设计与核心代码实现(85% 代码量)
- 设计 prompt + tool 描述,Computer Use 任务成功率从 41% 提升到 78%
- 自研操作回放系统,失败 case 可一键复现,大幅提升调试效率

**项目成果**:
- 内测 30 位分析师,日均节省 1.8 小时/人,折合月度节省 1200+ 工时
- 公司内部 PMF 验证后启动商业化,首批签约客户 8 家,ARR 80W
- 在公司技术大会做主题分享,推动 Agent 方向立项扩招

## 范例 4:垂直领域 RAG Agent

**项目名称**:法律文书智能检索与起草 Agent
**项目周期**:2024.10 - 2025.06 | **角色**:RAG 算法工程师(4 人团队)

**项目背景**:
律所合伙人起草合同/诉状时,需查阅历史相似案例与法条,平均耗时 4 小时。
构建专业级法律 Agent,目标将起草时间压缩到 45 分钟。

**技术方案**:
- 数据层:清洗 60W+ 裁判文书 + 12W+ 合同模板,使用 **LlamaParse** 处理 PDF 表格与图片
- 检索层:**GraphRAG** 构建案例-法条-当事人知识图谱,关联检索准确率比纯向量提升 34%
- 生成层:基于 **Claude Sonnet 4.6** + 结构化输出,起草内容自动引用法条出处
- 反幻觉:每条法条引用必须命中知识库,未命中则拒答;律师反馈准确率 96%
- 工程化:**LiteLLM** 做模型路由,简单查询走 Haiku,复杂起草走 Opus,综合成本下降 58%

**个人贡献**:
- 主导 GraphRAG 知识图谱设计(实体识别、关系抽取、图嵌入)
- 设计法条引用验证机制,把幻觉率从 18% 压到 1.2%
- 输出 2 篇内部技术博客,被引用作为新人培训材料

**项目成果**:
- 律所内部 200+ 律师使用,起草时间从 4 小时降到 38 分钟
- 客户案件办理周期缩短 22%,律所人均产能提升 35%
- 产品获省级司法局推荐,落地 3 家合作律所

## 通用句式模板

**项目背景类**(描述痛点):
- "公司 [部门] 每日 [动作],其中 [%] 为 [问题],造成 [损失数字]"
- "现有方案 [技术] 在 [场景] 下 [问题],无法满足 [业务诉求]"

**技术方案类**(突出选型):
- "基于 [框架] 设计 [架构],包含 [N] 个核心模块"
- "对比 [方案 A / B / C],最终选择 [方案] 因为 [原因]"
- "通过 [技术],把 [指标] 从 [X] 提升到 [Y]"

**个人贡献类**(明确 ownership):
- "主导 [模块] 设计与实现,代码贡献占 [%]"
- "独立解决 [疑难问题],方案被团队采纳"
- "推动 [流程改进],团队效率提升 [%]"

**成果类**(数字优先):
- "上线 [N] 个月,日均 [QPS / 用户数 / 调用量]"
- "[业务指标] 提升 [%] / 降低 [%] / 节省 [金额]"
- "P99 延迟 [Xms],可用性 [99.X%]"
