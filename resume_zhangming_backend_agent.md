### 项目二:智能运维 Agent「OpsCopilot」
**2024.01 - 2024.10** | **Agent 后端 Tech Lead(4 人团队)**

**项目背景**
> 公司 SRE 团队每日处理 200+ 告警,80% 为重复模式(数据库慢查询、容器 OOM、网络抖动),
> 平均 MTTR(平均修复时间)45 分钟,深夜告警严重影响 oncall 体验。
> 需构建能自动诊断、给出修复建议、必要时执行修复的运维 Agent。

**技术方案**
- 基于 **LangGraph** 设计 ReAct 模式 Agent,核心 4 个节点:告警归类 → 数据采集 → 根因分析 → 修复执行
- 自研 **Ops MCP Server**,封装 Prometheus 查询、ELK 日志检索、K8s pod 操作、数据库慢查询分析等 18 个工具
- **知识库 RAG**:导入公司近 3 年 Runbook + 历史故障复盘(共 1200 篇),使用 **GraphRAG** 构建
  故障-原因-处置关系图,根因匹配准确率比纯向量提升 **31%**
- **安全机制**:所有写操作(重启 pod、kill 慢查询、扩容)走 **人工 approval 流**,只读诊断全自动
- 接入 **PagerDuty / 飞书机器人**,告警触发后 Agent 直接在群里 @ oncall 给出诊断报告

**个人贡献**
- 主导 Agent 整体设计,推动从 "纯诊断" 演进到 "诊断 + 受控修复" 的产品形态
- 独立实现 Ops MCP Server 框架,被复用到后续 2 个内部 Agent 项目
- 设计 **三级置信度评估**(LLM-judge + 规则 + 历史相似度),低置信度自动转人工,误处置率 < **0.3%**

**项目成果**
- 📊 上线 6 个月,覆盖 95% 的告警类型,自动诊断准确率 **87%**
- ⚡ MTTR 从 45 分钟降到 **9 分钟**,自动修复占比 23%(其余给出可一键执行的建议)
- 💤 深夜(00:00-06:00)告警 oncall 介入次数下降 **78%**,SRE 团队 NPS 从 32 提升到 71
- 📝 项目方案在内部技术大会分享,被 3 个兄弟部门复刻

### 项目三:亿级 QPS 支付网关重构
**2021.03 - 2023.08** | **架构负责人(12 人团队)** | 公司支付中台核心项目

> 一次大促事故影响营收约 ¥2000W。需要架构级重构。

**技术方案**
- **架构升级**:Spring Cloud 单体 → **Go + 自研 RPC 框架**,核心路径性能提升 **5x**

  实现 **自适应熔断**,基于响应时间动态调整阈值
