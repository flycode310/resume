## 一、Go 语言必考题（基础门槛）

> 这家公司 Go 是主力语言，**这块答不好直接挂**。

### 1.1 GMP 模型与 goroutine 调度

**预测问题**：Go 的 GMP 模型是什么？goroutine 是怎么调度的？

**回答思路**：
- **G** (Goroutine)：协程，包含栈和程序计数器
- **M** (Machine)：操作系统线程，由 OS 调度
- **P** (Processor)：逻辑处理器，维护本地 runqueue，是 G 和 M 之间的桥梁
- **调度流程**：M 必须绑定 P 才能执行 G；P 维护本地队列（LRQ），空时从全局队列（GRQ）或其他 P 偷（work stealing）
- **抢占式调度**：Go 1.14 之前是协作式（函数调用时检查），之后加入基于信号的抢占
- **实战经验**：[讲一个你调过的 goroutine 泄露 / 调度延迟的 case]

### 1.2 channel 原理

**预测问题**：无缓冲 channel 和有缓冲 channel 有什么区别？channel 底层是什么结构？

**回答思路**：
- 底层结构 `hchan`：包含 buf（环形缓冲区）、sendx/recvx（索引）、sendq/recvq（等待队列）、lock（互斥锁）
- **无缓冲**：发送和接收必须同时就绪，否则 goroutine 进入等待队列
- **有缓冲**：缓冲区未满时发送不阻塞，未空时接收不阻塞
- **关闭后行为**：关闭后接收方能读完剩余数据，再读返回零值和 ok=false；**关闭后再发送会 panic**
- **实战经验**：[讲一个 channel 误用导致 deadlock 的 case，比如在 defer 中关闭 channel]

### 1.3 内存管理与 GC

**预测问题**：Go 的 GC 是怎么工作的？什么时候会触发 STW？

**回答思路**：
- **三色标记清除法**：从根对象出发，白（未访问）→ 灰（已发现未扫描）→ 黑（已扫描）；并发执行
- **混合写屏障**（Go 1.8+）：保证并发标记的正确性
- **触发时机**：默认 GC 触发阈值 = 上次 GC 后存活堆内存 × GOGC（默认 100）
- **STW 优化**：Go 1.14 后 STW 时间降到 100 微秒以下；大多数情况下 P99 < 1ms
- **实战经验**：[讲一个内存泄漏 case，比如 goroutine 引用导致对象无法回收]

### 1.4 context 包

**预测问题**：context 怎么用？什么场景会 cancel？

**回答思路**：
- 四个方法：`WithCancel / WithTimeout / WithDeadline / WithValue`
- **传播链路**：父 context cancel → 所有子 context 自动 cancel
- **使用原则**：作为函数第一个参数传递（"context should be the first parameter"）
- **典型场景**：HTTP 请求超时、RPC 调用链传递、goroutine 生命周期管理
- **坑**：不要把 context 放在 struct 里传递；不要传 nil（用 context.TODO() 占位）

### 1.5 并发安全

**预测问题**：sync.Mutex、sync.RWMutex、sync.Map 各适用什么场景？

**回答思路**：
- **Mutex**：写多读少，临界区短的场景
- **RWMutex**：读多写少，多个读可并发，写独占
- **sync.Map**：读多写少 + key 稳定（如配置、路由表），底层 read-only + dirty 双 map
- **atomic 包**：单个变量的原子操作，比 mutex 轻量
- **实战经验**：[讲一个选错锁导致性能问题的 case]

### 1.6 性能调优

**预测问题**：Go 程序 CPU 飙高或内存占用大，你怎么排查？

**回答思路**：
- **pprof**：`net/http/pprof` 暴露 profiling 接口；`go tool pprof` 分析
- **火焰图**：brendangreve 的 FlameGraph，把 CPU 耗时按调用栈展开
- **trace**：go tool trace 看调度延迟、GC、网络 I/O
- **典型流程**：CPU 飙高 → pprof top → 看热点函数 → 看调用栈 → 定位代码 → 修复
- **实战经验**：必须准备一个真实 case，**有数字**（如「通过 pprof 定位到一个死循环，把 QPS 从 800 提升到 5000」）

### 1.7 其他高频考点

- **slice 底层**：ptr + len + cap；append 时扩容策略（cap < 1024 时翻倍，否则增长 25%）
- **map 底层**：哈希桶 + 链表/红黑树（Go 1.8 后冲突超过 8 个用红黑树）
- **interface 底层**：eface（空接口）和 iface（非空接口）；类型断言原理
- **defer 机制**：基于链表实现，注册时按 LIFO 执行
- **error vs panic**：error 用于可预期的错误，panic 用于真正异常的情况

## 二、分布式系统必考题

### 2.1 限流算法

**预测问题**：常见的限流算法有哪些？你的项目怎么用的？

**回答思路**：
- **计数器**：固定窗口 / 滑动窗口
- **漏桶**：恒定速率流出，平滑流量
- **令牌桶**：允许突发流量（推荐），可用 Redis + Lua 实现分布式
- **Sentinel / Sentinel-Go**：阿里开源，支持限流、熔断、降级
- **实战经验**：你的 2025 司机 FTO 项目限流怎么做的？

### 2.2 分布式事务

**预测问题**：分布式事务有哪些方案？分别有什么 trade-off？

**回答思路**：
- **2PC**：强一致但同步阻塞，协调者单点
- **3PC**：引入超时和预提交，减少阻塞
- **TCC**：Try-Confirm-Cancel，性能好但业务侵入
- **Saga**：长事务拆分，每个子事务有补偿动作；分编排（Orchestration）和编制（Choreography）
- **本地消息表**：最终一致，实现简单
- **实战经验**：你的 2023 花小猪撮合流程怎么保证撮合 + 计费一致性？

### 2.3 一致性协议

**预测问题**：Raft 和 Paxos 有什么区别？为什么工业界多用 Raft？

**回答思路**：
- **Paxos**：理论奠基但难以理解；Multi-Paxos 实用化
- **Raft**：易理解，分 Leader 选举、日志复制、安全性三部分
- **Raft 优化**：PreVote（防止分区恢复后的无效选举）、Batch（批量提交）、Pipeline
- **etcd 实现**：基于 Raft 的 KV 存储，WAL + snapshot
- **实战经验**：你用过 etcd / consul 吗？什么场景？

### 2.4 微服务架构

**预测问题**：你们的微服务架构是怎么治理的？服务之间怎么通信？

**回答思路**：
- **服务注册发现**：etcd / consul / nacos
- **服务通信**：gRPC（推荐）/ HTTP
- **服务网格**：Istio（了解即可）
- **API 网关**：Kong / Envoy / 自研
- **可观测性**：OpenTelemetry + Prometheus + Grafana + Jaeger
- **实战经验**：你的 30+ 跨团队协调是怎么做的？规范、流程、工具？

### 2.5 单元化架构

**预测问题**：什么是单元化架构？你们的业务为什么用单元化？

**回答思路**：
- **概念**：按用户 ID 把流量路由到指定单元（机房），单元内自包含
- **优势**：跨机房调用降到最低、机房级故障 30 秒自动切流、容量水平扩展
- **挑战**：数据同步、跨单元查询、全局 ID 生成
- **实战经验**：你的 2025 司机 FTO 项目有没有单元化设计？

## 三、Agent / LLM 必考题（最重要）

> 这是 JD 的核心区分度，**答不好被当成普通后端**。

### 3.1 Agent 框架对比

**预测问题**：LangGraph、Anthropic Agent SDK、AutoGen 各有什么优劣？你们为什么选这个？

**回答思路**：
- **LangGraph**：状态机式编排，可视化好，Human-in-the-loop 强；适合复杂多步工作流
- **Anthropic Agent SDK**：官方出品，与 MCP 原生集成；简洁，适合工具调用场景
- **AutoGen v0.4**：异步事件驱动，多 Agent 对话；适合研究类
- **CrewAI**：角色扮演式；上手快但灵活度低
- **选型决策**：业务需要多步工具调用 + 复杂反思 → 选 LangGraph；想快速集成 MCP → Anthropic Agent SDK；多 Agent 协作 → AutoGen
- **实战经验**：**你必须能讲清楚选 Anthropic Agent SDK 的具体原因**，如「MCP 原生集成避免额外胶水代码，团队学习成本低」

### 3.2 ReAct 原理

**预测问题**：ReAct 是什么？为什么 Agent 普遍用 ReAct？

**回答思路**：
- **ReAct = Reasoning + Acting**：推理（Chain-of-Thought）+ 行动（工具调用）交替
- **每一步**：思考 → 行动 → 观察 → 再思考
- **优势**：可解释、可调试、支持反思重试
- **局限**：长链路 token 消耗大、可能出现循环
- **优化**：限制最大步数、引入 confidence 评分、低置信度自动转人工
- **实战经验**：你的 harness 风格 tool-use 循环具体怎么实现的？plan → act → observe → reflect 每步做什么？

### 3.3 Function Calling 协议

**预测问题**：Function Calling 的协议是怎样的？JSON Schema 怎么写？

**回答思路**：
- **协议流程**：用户问题 → 模型判断是否需要调用工具 → 返回工具名+参数 → 执行工具 → 把结果喂回模型 → 模型继续推理
- **工具描述**：JSON Schema 定义参数，包括 type / description / required / enum
- **错误处理**：工具调用失败时怎么重试？参数校验失败怎么提示模型？
- **实战经验**：你的 MCP Server 怎么定义工具？参数 schema 怎么写？

### 3.4 MCP 协议

**预测问题**：MCP 是什么？和 Function Calling 有什么区别？

**回答思路**：
- **MCP = Model Context Protocol**：Anthropic 提出的标准协议，2024 年开源
- **三大原语**：Resources（资源）、Prompts（提示模板）、Tools（工具）
- **三种 transport**：stdio（本地进程）、SSE（服务端推送）、HTTP（远程调用）
- **vs Function Calling**：FC 是单次工具调用，MCP 是标准化工具提供方式，可被发现、可重用
- **架构**：MCP Host（Claude Desktop / Cursor）+ MCP Client + MCP Server
- **实战经验**：你的 6 个 MCP Server 怎么实现的？stdio 还是 SSE？为什么？

### 3.5 RAG 架构设计

**预测问题**：RAG 的核心环节有哪些？召回不准怎么优化？

**回答思路**：
- **核心环节**：文档解析 → 分块（chunking）→ Embedding → 检索 → 重排 → 生成
- **召回优化**：
  - Chunking 策略：固定长度 vs 语义分块 vs AST 分块（代码）
  - 混合检索：BM25 + 向量检索
  - HyDE：假设性文档嵌入
  - Query 改写：Multi-Query / Step-Back
- **重排**：bge-reranker / Cohere Rerank，Top-K 候选 → Top-N 最终
- **评估**：Hit Rate / MRR / NDCG
- **实战经验**：你的代码检索为什么用 AST chunk + 语义 chunk 双索引？

### 3.6 向量数据库选型

**预测问题**：Milvus、Qdrant、PGVector 怎么选？

**回答思路**：
- **Milvus**：分布式、亿级规模、生产首选；学习曲线陡
- **Qdrant**：Rust 实现、性能好、易部署；中等规模首选
- **PGVector**：PostgreSQL 插件、运维简单、规模有限
- **Pinecone / Weaviate**：云托管 / 文档型
- **选型决策**：规模（亿级 vs 百万级）、运维能力、查询延迟要求
- **实战经验**：你为什么选 Qdrant？

### 3.7 Prompt 工程

**预测问题**：你们怎么管理 Prompt？版本、回滚、A/B 测试怎么做？

**回答思路**：
- **版本管理**：Prompt CR + Git 管理 + 灰度发布
- **A/B 测试**：按租户 / 用户分流，对比成功率 / 用户反馈
- **评估**：LLM-as-Judge + 人工抽样 + golden set
- **工具**：Promptfoo（CI 集成）/ PromptLayer（托管）/ LangSmith（一体化）
- **结构化输出**：JSON Mode / Pydantic schema / Tool Use
- **实战经验**：你的 Prompt Review 机制具体怎么做？

### 3.8 Agent 评估

**预测问题**：怎么评估 Agent 的效果？怎么知道改了一个 Prompt 后效果变好了？

**回答思路**：
- **Golden set**：构造 200-500 个真实 case，作为评估基准
- **评估指标**：任务完成率、工具调用准确率、平均步数、token 消耗
- **LLM-as-Judge**：用更强的模型（Opus 4.7）评估弱模型（Sonnet 4.6）的输出
- **A/B 测试**：新旧版本分流对比
- **Promptfoo**：CI 集成，每次 Prompt 变更自动跑 golden set
- **实战经验**：你的 golden set 怎么构建的？跑了多少 case？

### 3.9 LLM 成本治理

**预测问题**：你们的 token 成本怎么降的？

**回答思路**：
- **Prompt Caching**：Claude Sonnet 4.6 / Haiku 4.5 已支持，缓存系统提示词降本 60%+
- **语义缓存**：相同问题直接返回历史结果，命中率 30-40%
- **模型路由**：简单任务走 Haiku，复杂任务走 Sonnet / Opus
- **批处理**：非实时任务批量提交
- **流式输出**：避免超时重试
- **实战经验**：必须准备真实数字（命中率、降本百分比）

### 3.10 LLM Gateway 设计

**预测问题**：如果要你设计一个 LLM Gateway，会考虑什么？

**回答思路**：
- **核心能力**：模型路由、限流、降级、缓存、token 计量、可观测
- **架构**：上游业务 → Gateway → 多模型后端（Claude / GPT / Qwen）
- **关键模块**：
  - 路由策略：按任务类型 / 成本 / 延迟
  - 限流：按租户 / 用户 / API Key
  - 降级：主模型失败 → 备用模型 → 兜底回答
  - 缓存：语义缓存 + 精确缓存
  - 可观测：trace_id 全链路、prompt + response 记录
- **实战经验**：你的 LLM Gateway 是怎么实现的？性能指标？
